  - app.json infoPlist: Face ID / Touch ID / location (foreground +
    background) / camera / photo library usage strings present.
  - PrivacyInfo.xcprivacy: NOT present.

Actions Taken:
  - None. Wiring a PrivacyInfo manifest into a managed Expo build requires
    (a) an inventory of which "Required Reason APIs" the app actually uses
    (UserDefaults, file timestamps, system boot time, disk space) — which
    depends on the exact set of native modules pulled in by Expo SDK 54 +
    your plugins, and (b) either a config plugin or migration to
    expo-build-properties privacyManifests config. Producing a stub
    without that audit creates false comfort and can cause App Store
    rejections if it lists wrong reasons.

Remaining Gaps:
  - PrivacyInfo.xcprivacy missing. Apple has been enforcing this since
    May 2024 for new + updated apps using "Required Reason APIs".
  - Privacy nutrition label answers in App Store Connect (data types
    collected, linked-to-user vs not, used-for-tracking) are not tracked
    in-repo. Recommend a docs/privacy-disclosures.md mirroring the App
    Store Connect questionnaire so it's reviewable in code review.
  - Google Play Data Safety form not mirrored in repo either.

Notes / Recommended Next Steps:
  - Inventory native modules used (run `npx expo config --type prebuild`
    to dump the resolved native config). Cross-reference against Apple's
    Required Reason API list:
    https://developer.apple.com/documentation/bundleresources/privacy_manifest_files/describing_use_of_required_reason_api
  - Add `expo-build-properties` to plugins in app.config.js with
    ios.privacyManifests populated by the inventory above.
  - Mirror the privacy nutrition label answers as docs/privacy-disclosures.md.
